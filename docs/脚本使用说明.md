# Scripts 脚本使用说明

本文档说明项目中各个启动脚本的用途和使用方法。

## 当前脚本（根目录）

这些脚本目前位于项目根目录，存在冗余和混乱。未来会迁移到 `scripts/` 目录。

### start-all.sh
**用途**：启动完整开发环境（数据库 + Django + 网关 + Frontend）  
**使用**：`./start-all.sh`  
**覆盖范围**：
- PostgreSQL + Redis
- Django 后端服务
- APISIX/Traefik 网关
- Flutter 前端（如果支持）

### start-dev.sh
**用途**：启动开发模式（轻量级）  
**使用**：`./start-dev.sh`  
**与 start-all.sh 的区别**：可能仅启动必需的服务（数据库 + Django）

### start-backend.sh
**用途**：仅启动后端服务（Django）  
**使用**：`./start-backend.sh`  
**前置条件**：数据库和 Redis 已运行
**适用场景**：
- 数据库已在运行（例如之前启动）
- 需要快速重启 Django
- 调试 Django 特定问题

### start-frontend.sh
**用途**：仅启动前端（Flutter）  
**使用**：`./start-frontend.sh`  
**前置条件**：Django 后端已运行

### start-apisix.sh
**用途**：仅启动 APISIX 网关  
**使用**：`./start-apisix.sh`  
**用途**：建立 API 网关层，转发请求到后端服务

### stop-backend.sh
**用途**：停止后端服务  
**使用**：`./stop-backend.sh`

### stop-apisix.sh
**用途**：停止 APISIX 网关  
**使用**：`./stop-apisix.sh`

## 脚本目录（scripts/）

当前位于 `scripts/` 目录的脚本：

### start-local-all.sh
**用途**：启动完整本地开发环境  
**使用**：`./scripts/start-local-all.sh`  
**相当于**：根目录的 `start-all.sh`

### start-local-backend.sh
**用途**：启动本地后端（开发模式）  
**使用**：`./scripts/start-local-backend.sh`  
**特点**：可能配置热重载

### start-local-frontend.sh
**用途**：启动本地前端开发服务器  
**使用**：`./scripts/start-local-frontend.sh`

### start-prod.sh
**用途**：启动生产环境配置  
**使用**：`./scripts/start-prod.sh`  
**注意**：
- 使用生产数据库配置
- DEBUG = False
- 性能优化

## 问题分析

### 当前存在的问题

1. **脚本冗余**
   - 根目录和 `scripts/` 目录有重复的脚本
   - 同名脚本可能有不同的功能
   - 难以维护

2. **缺少细粒度控制**
   - 无法单独启动数据库
   - 无法只启动 Django + 数据库（不启动网关）
   - 无法灵活组合服务

3. **缺少初始化脚本**
   - 没有数据库初始化脚本（迁移、种子数据）
   - 没有数据清理脚本
   - 没有示例数据加载脚本

4. **缺少文档**
   - 脚本内没有说明注释
   - 不清楚每个脚本的入口和依赖
   - 新人容易弄错启动顺序

5. **网关配置混乱**
   - 同时有 Traefik 和 APISIX 脚本
   - 不清楚应该使用哪个
   - 迁移状态不明确

### 各脚本详细说明

#### db/init.sh
```bash
#!/bin/bash
# 初始化数据库：运行迁移 + 种子数据
# 用法: ./scripts/db/init.sh
# 说明:
#   1. 应用所有未应用的迁移
#   2. 创建超级用户（如果不存在）
#   3. 加载示例数据（如果存在）
```

#### db/clean.sh
```bash
#!/bin/bash
# 清理数据库：重置为初始状态
# 用法: ./scripts/db/clean.sh
# 警告: 将删除所有数据！使用前请确认
# 步骤:
#   1. 删除所有数据（flush）
#   2. 重新运行迁移
#   3. 创建新的超级用户
```

#### backend/hot-reload.sh
```bash
#!/bin/bash
# 启动 Django 开发服务器（热重载模式）
# 用法: ./scripts/backend/hot-reload.sh
# 特性:
#   - 文件变化时自动重启
#   - 保存代码后立即生效
#   - 比 restart 容器更快
```

#### backend/django-with-db.sh
```bash
#!/bin/bash
# 启动 Django + PostgreSQL + Redis（完整开发栈）
# 用法: ./scripts/backend/django-with-db.sh
# 功能:
#   1. 启动 PostgreSQL（如果未运行）
#   2. 启动 Redis（如果未运行）
#   3. 运行数据库迁移
#   4. 启动 Django 开发服务器
```

#### gateway/apisix-only.sh

## 当前推荐用法

### 首次运行

```bash
# 1. 启动完整环境
docker-compose -f infra/docker-compose-apisix.yml up -d

# 2. 初始化数据库
docker-compose -f infra/docker-compose-apisix.yml exec django python manage.py migrate

# 3. 创建超级用户
docker-compose -f infra/docker-compose-apisix.yml exec django python manage.py createsuperuser

# 4. 访问验证
# - API Gateway: http://localhost:9080/health/
# - Admin Panel: http://localhost:9080/admin/
# - Dashboard: http://localhost:9000/
```

### 日常开发

```bash
# 启动开发环境
docker-compose -f infra/docker-compose-apisix.yml up -d

# 查看日志
docker-compose -f infra/docker-compose-apisix.yml logs -f django

# 停止服务
docker-compose -f infra/docker-compose-apisix.yml down
```

### 修改数据库模型后

```bash
# 创建迁移
docker-compose -f infra/docker-compose-apisix.yml exec django python manage.py makemigrations

# 应用迁移
docker-compose -f infra/docker-compose-apisix.yml exec django python manage.py migrate
```

## 常见问题

**Q: 应该运行哪个脚本？**
A: 推荐使用 `docker-compose` 直接运行：
```bash
docker-compose -f infra/docker-compose-apisix.yml up -d
```

**Q: 脚本执行失败，怎么办？**
A: 
1. 查看错误日志
2. 确认 Docker 和 Docker Compose 已安装
3. 确认有足够的磁盘空间和权限

**Q: 如何仅启动数据库？**
A: 当前不支持，需要通过 `docker-compose` 手动操作：
```bash
docker-compose -f infra/docker-compose.yml up -d postgres redis
```
