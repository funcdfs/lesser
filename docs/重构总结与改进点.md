#### 3. 架构设计

**Frontend 分层架构**：
```
domain/    - 纯业务模型（无框架依赖）
data/      - 数据层（API、Repository）
feature/   - 功能模块（UI + State）
app/       - 应用配置（路由、主题）
shared/    - 跨功能共享资源
```

**Django 应用结构规范**：
```
app_name/
├── models/        - 按功能分文件
├── serializers/   - 按功能分文件
├── views/         - 按功能分文件
├── urls.py        - 路由注册
├── permissions.py - 权限检查
└── ...
```

## 二、当前存在的主要问题

### 1️⃣ Frontend 工程噪音

| 问题 | 影响 | 优先级 |
|------|------|--------|
| lib/ 目录结构不清晰，分层混乱 | 难以定位代码，协作困难 | 🔴 高 |
| UI 组件划分不够细腻，文件庞大 | 代码复用性差，维护困难 | 🔴 高 |
| Theme 管理分散，样式定义不统一 | 界面不一致，修改困难 | 🟡 中 |
| 没有清晰的分层文档 | 新人上手困难，约束不清 | 🟡 中 |

**优化建议**：
- [ ] 按提案创建新的 `lib/` 目录结构（domain/data/feature/app）
- [ ] 保留旧代码在 `lib/old/` 目录，逐步迁移
- [ ] 统一 Theme 定义在 `app/theme/`
- [ ] 为 Flutter 项目添加分层架构指南（已完成 📄）

### 2️⃣ Django 缺乏文档和规范

| 问题 | 影响 | 优先级 |
|------|------|--------|
| 没有清晰的项目说明 | 新人不理解 Django 结构 | 🔴 高 |
| App 内部平面化，难以扩展 | 代码混乱，难以维护 | 🔴 高 |
| settings 配置无说明 | 难以理解各配置的作用 | 🟡 中 |
| 没有规范化的命名 | 代码风格不统一 | 🟡 中 |

**优化建议**：
- [x] 添加 `README.md` 详细说明（已完成 📄）
- [x] 为 `settings.py` 添加分组注释（已完成 ✅）
- [ ] 为每个 app 添加 `README.md` 说明

### 3️⃣ 基础设施混乱

| 问题 | 影响 | 优先级 |
