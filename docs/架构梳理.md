## 核心设计理念

### 读写分离

| 服务 | 职责 | 一致性 |
|------|------|--------|
| Feed Write Service | 发帖、删除、大V判定 | 强一致性 |
| Feed Read Service | Timeline 聚合、Hybrid 合并 | 最终一致性 |

### Hybrid Timeline 策略

```
┌─────────────────────────────────────────────────────────────────────────┐
│                     Hybrid Timeline 读取策略                             │
└─────────────────────────────────────────────────────────────────────────┘

普通用户（粉丝 < 阈值）：
  └── 写扩散（Push）：发帖时异步写入所有粉丝的 Timeline

大V用户（粉丝 >= 阈值）：
  └── 读扩散（Pull）：读取时实时拉取大V最新帖子

Timeline 合并流程：
  1. 从 Redis 拉取用户 Timeline（已扩散的普通用户帖子）
  2. 从 Redis 拉取关注的大V最新帖子
  3. 合并排序，返回最终 Feed
```

### Sequence 机制（Chat）

```
┌─────────────────────────────────────────────────────────────────────────┐
│                     消息序列号分配                                        │
└─────────────────────────────────────────────────────────────────────────┘

Sequence Allocator 职责：
  ├── 为每个会话分配递增的 SeqID
  ├── 保证消息顺序性
  └── 支持增量同步（客户端只拉取 > lastSeqID 的消息）

流程：
  1. ChatSvc 请求 SeqSvc 分配 SeqID
  2. SeqSvc 从 Redis INCR 获取下一个序列号
  3. 消息携带 SeqID 写入 Postgres
  4. 客户端通过 SeqID 实现增量同步
```

## 服务间通信原则

### 强一致性 vs 最终一致性

| 场景 | 一致性要求 | 通信方式 | 说明 |
|------|-----------|---------|------|
| 发帖/删帖 | 强一致性 | gRPC 同步 | 写入 Postgres + Redis |
| 消息发送 | 强一致性 | gRPC 同步 | SeqID 分配 + 持久化 |
| 计数更新 | 强一致性 | gRPC 同步 | 点赞/评论/转发计数 |
| 权限检查 | 强一致性 | gRPC 同步 | 内容存在性、成员权限 |
| Timeline 扩散 | 最终一致性 | RabbitMQ | 普通用户 Fanout |
| 通知推送 | 最终一致性 | RabbitMQ | 延迟几秒可接受 |
| 搜索索引 | 最终一致性 | RabbitMQ | 索引延迟可接受 |
| 审计统计 | 最终一致性 | RabbitMQ | 异步记录 |

### gRPC 同步调用（强一致性）

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        服务间 gRPC 调用关系                              │
└─────────────────────────────────────────────────────────────────────────┘

Feed Write ──gRPC──► Postgres (Write Post)
     │
     └──gRPC──► Redis (Author Posts / BigV Flag)

Feed Read ──gRPC──► Redis (User Timeline + BigV Posts)
     │
     └──gRPC──► Postgres (Fallback)

Chat Core ──gRPC──► Sequence Allocator (Alloc SeqID)
     │
     ├──gRPC──► Postgres (Durability)
     │
     └──gRPC──► Sync Service (Realtime Push)

Sync Service ──gRPC Stream──► App (多端同步)
```

### RabbitMQ 异步消息（最终一致性）

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        RabbitMQ 事件流                                   │
└─────────────────────────────────────────────────────────────────────────┘

事件生产者                    Exchange              事件消费者
─────────────────────────────────────────────────────────────────────────
Feed Write ──► [post_created] ──► Timeline Worker
                                      └── 普通用户 Fanout 到 Redis

Feed Write ──► [post_created] ──► ElasticSearch
               [post_deleted]        └── 更新搜索索引

Chat Core ──► [message_sent] ──► Notification Service
                                     └── 离线推送通知

Chat Core ──► [message_sent] ──► Audit Service
                                     └── 消息审计记录

Comment Service ──► [comment.created] ──► Notification Service
                    [comment.liked]           └── 评论通知

Interaction Service ──► [content.liked] ──► Notification Service
                        [content.reposted]       └── 点赞/转发通知

User Service ──► [user.followed] ──► Notification Service
                                        └── 关注通知
```

## 服务职责

### Feed Write Service

内容写入服务（强一致性）：
- 发帖（Story/Short/Article）
- 删帖
- 大V判定（粉丝数 >= 阈值）
- 写入 Postgres + Redis
- 发布 `post_created` 事件

```
Feed Write ──► Postgres (持久化)
     │
     ├──► Redis (作者帖子列表 / 大V标记)
     │
     └──► RabbitMQ (post_created 事件)
```

### Feed Read Service

Timeline 聚合服务（Hybrid 策略）：
- 拉取用户 Timeline（已扩散的帖子）
- 拉取关注的大V最新帖子
- 合并排序
- Postgres Fallback（Redis 未命中时）

```
Feed Read ──► Redis (User Timeline)
     │
     ├──► Redis (BigV Recent Posts)
     │
     └──► Postgres (Fallback)
```

### Chat Core Service

聊天核心服务（状态机 + gRPC Stream）：
- 会话管理（创建/查询）
- 消息发送（SeqID 分配 + 持久化）
- 消息状态机（发送中/已发送/已读）
- 实时推送（通过 Sync Service）

```
Chat Core ──► Sequence Allocator (SeqID)
     │
     ├──► Postgres (消息持久化)
     │
     ├──► Sync Service (实时推送)
     │
     └──► RabbitMQ (离线通知/审计)
```

### Sequence Allocator

序列号分配服务：
- 为每个会话分配递增 SeqID
- Redis INCR 保证原子性
- 支持增量同步

```
Sequence Allocator ──► Redis (INCR conversation:{id}:seq)
```

### Sync Service

多端同步服务：
- 维护客户端 gRPC Stream 连接
- 实时推送新消息
- 支持多设备同步
- 增量同步（基于 SeqID）

```
Sync Service ──gRPC Stream──► App (Phone)
     │
     └──gRPC Stream──► App (PC)
```

### Timeline Worker

异步扩散服务：
- 消费 `post_created` 事件
- 普通用户 Fanout（写入粉丝 Timeline）
- 写入 Redis Timeline

```
RabbitMQ ──► Timeline Worker ──► Redis (Fanout to Followers)
```

### Audit / Stats Service

审计统计服务：
- 消息审计记录
- 内容审计记录
- 统计指标收集
- 异常行为检测

### Notification Service

通知服务：
- 消费各类事件
- 创建通知记录
- 离线推送（APNs/FCM）
- 未读数统计

## Chat 双向流

### 消息流程

```
客户端                    服务端
   │                        │
   │── Subscribe(convId) ──>│  订阅会话
   │<── Subscribed ─────────│
   │                        │
   │── SendMessage ────────>│  发送消息
   │                        │  1. Alloc SeqID
   │                        │  2. Write Postgres
   │                        │  3. Push via SyncSvc
   │<── NewMessage ─────────│  广播给所有订阅者
   │                        │
   │── Typing ─────────────>│  正在输入
   │<── TypingIndicator ────│
   │                        │
   │── Ping ───────────────>│  心跳
   │<── Pong ───────────────│
   │                        │
   │── Sync(lastSeqID) ────>│  增量同步
   │<── Messages[] ─────────│  返回 > lastSeqID 的消息
```

### 事件类型

客户端事件：`Subscribe`, `Unsubscribe`, `SendMessage`, `Typing`, `Ping`, `Sync`

服务端事件：`NewMessage`, `MessageRead`, `TypingIndicator`, `Pong`, `Error`, `SyncResponse`

### 多端同步

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        多端增量同步                                       │
└─────────────────────────────────────────────────────────────────────────┘

场景：用户在手机发消息，PC 端实时收到

1. 手机发送消息
   Phone ──► Chat Core ──► SeqID=100 ──► Postgres

2. 实时推送
   Chat Core ──► Sync Service ──► PC (gRPC Stream)

3. PC 离线后重连
   PC ──► Sync(lastSeqID=95) ──► Chat Core
   Chat Core ──► 返回 SeqID > 95 的消息
```

### 重连机制

- 指数退避：1s → 2s → 4s → ... → 30s（最大）
- 重连后自动重新订阅之前的会话
- 增量同步：携带 lastSeqID 拉取缺失消息
- 心跳间隔：30s，超时：10s
