### 核心功能

1. **`pkg/broker` (RabbitMQ 引擎)**
   - `Worker` 结构体封装 `amqp091-go` 连接
   - 自动重连机制（指数退避算法，最大 30 秒）
   - `Consume(queueName, handler)` 核心方法
   - 优雅停机：监听 `SIGINT/SIGTERM`，确保消息处理完成后关闭
   - 从消息头自动提取 `trace_id` 注入 context

2. **`pkg/database` (PostgreSQL)**
   - 连接池配置（MaxOpenConns, MaxIdleConns, ConnMaxLifetime）
   - `MustInitDB` 方法，连接失败时 panic
   - 健康检查方法

3. **`pkg/logger` (Zap 封装)**
   - 生产环境：JSON 格式
   - 开发环境：Console 格式（带颜色）
   - 自动注入 `trace_id` 和 `user_id` 到日志上下文
   - 符合 `docs/logging-guidelines.md` 规范

4. **`pkg/app` (应用生命周期)**
   - 统一的 `App` 结构体管理所有组件
   - 支持注册自定义组件
   - 统一的启动和关闭顺序
   - 从环境变量自动读取配置

5. **`pkg/config` (配置管理)**
   - `GetEnv`, `GetEnvInt`, `GetEnvBool`, `GetEnvDuration`
   - `MustGetEnv` 必需配置检查

### 使用示例

重构后的 Worker 代码：

```go
// service/post_worker/cmd/worker/main.go
package main

import (
    "context"
    "github.com/lesser/pkg/app"
    "github.com/lesser/pkg/broker"
    "github.com/lesser/post_worker/internal/service"
)

func main() {
    ctx := context.Background()

    // 1. 初始化应用（自动连接 DB、配置日志）
    cfg := app.ConfigFromEnv("post-worker")
    application, err := app.New(cfg)
    if err != nil {
        panic(err)
    }

    // 2. 创建业务服务
    postSvc := service.NewPostService(application.DB())

    // 3. 配置队列消费
    brokerConfigs := []broker.Config{
        {Queue: "post.create", Handler: postSvc.HandleCreate},
        {Queue: "post.delete", Handler: postSvc.HandleDelete},
    }

    // 4. 启动（阻塞直到收到停止信号）
    if err := application.Run(ctx, brokerConfigs...); err != nil {
        panic(err)
    }
}
```

## 原始设计文档

这个 `pkg` 不是简单的代码拷贝，而是一个基于 **"组合优于继承"** 哲学的微服务引擎。它将处理所有高并发下的"脏活"（如断线自动重连、优雅停机、全链路日志追踪）。

### 给架构师的深度建议

* **共享 `api/` 目录**：除了 `pkg`，建议将所有的 `.proto` 文件生成的 Go 代码放在根目录的 `api/` 文件夹。这样 Gateway 和 5 个 Worker 引用的是同一套类型定义。
* **环境变量优先**：在 `pkg` 中内置一套基于 `os.LookupEnv` 的配置读取逻辑，让 Worker 启动时自动寻找 `MQ_URL`, `DB_URL` 等。
* **TraceID 传递**：在 `pkg/broker` 的封装中，务必从 RabbitMQ 的消息头（Headers）中提取 `trace_id` 并放入 `context.Context`。这样你通过 Refine 或 Flutter 发出的请求，在任何一个 Worker 的日志里都能被追踪到。
