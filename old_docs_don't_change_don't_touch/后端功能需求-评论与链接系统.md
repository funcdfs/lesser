## 1. 概述

### 1.1 系统架构

客户端采用独立的评论和链接系统，支持：

- **多级评论嵌套**：最多 30 层评论树结构
- **深层链接导航**：统一的 URL 格式和跨页面跳转
- **实时交互**：乐观更新、点赞、回复、删除
- **锚点定位**：快速跳转到 header/bottom 或指定评论

### 1.2 技术要求

- **存储方式**：推荐使用闭包表 (Closure Table) 存储评论树
- **分页方式**：基于游标的分页 (Cursor-based Pagination)
- **缓存策略**：评论数量、用户头像列表需要缓存
- **实时性**：支持乐观更新，失败时回滚

## 2. 评论系统 (Comment System)

### 2.1 核心功能

#### 2.1.1 评论入口数据

**功能描述**：提供评论入口所需的数据，用于渲染评论按钮

**数据结构**：
```dart
CommentEntryData {
  avatarUrls: List<String>,  // 最近评论者的头像 URL 列表（最多 5 个）
  count: int                 // 评论总数
}
```

**后端需求**：
- 查询指定目标（帖子/消息）的评论总数
- 返回最近 5 位评论者的头像 URL（去重，按时间倒序）
- 需要缓存优化，避免频繁查询

#### 2.1.2 加载评论列表

**功能描述**：加载指定目标的所有评论（根总览层）

**请求参数**：
```
targetId: string      // 目标 ID（帖子/消息 ID）
targetType: string    // 目标类型（post/message/article 等）
cursor?: string       // 分页游标（可选）
limit: int            // 每页数量（建议 20-50）
```

**响应数据**：
```dart
CommentListState {
  comments: List<CommentModel>,  // 评论列表
  pinnedComment?: CommentModel,  // 置顶评论（可选）
  totalCount: int,               // 评论总数
  hasMore: bool,                 // 是否有更多
  cursor?: string                // 下一页游标
}
```

**后端需求**：
- 按时间升序排列（最早的评论在前）
- 支持置顶评论（单独返回，不计入列表）
- 使用游标分页，避免深分页性能问题
- 返回每条评论的子孙数量（用于显示"查看回复"）

#### 2.1.3 加载子评论线程

**功能描述**：加载某条评论的所有回复（子层视图）

**请求参数**：
```
rootCommentId: string  // 根评论 ID
cursor?: string        // 分页游标
limit: int             // 每页数量
```

**响应数据**：
```dart
CommentListState {
  rootComment: CommentModel,     // 根评论
  comments: List<CommentModel>,  // 回复列表
  totalCount: int,               // 回复总数
  hasMore: bool,
  cursor?: string
}
```

**后端需求**：
- 查询指定评论的所有直接子评论
- 按时间升序排列
- 支持分页加载

#### 2.1.4 发表评论

**功能描述**：发表新评论或回复

**请求参数**：
```
targetId: string       // 目标 ID
targetType: string     // 目标类型
content: string        // 评论内容
replyToId?: string     // 回复目标 ID（可选）
```

**响应数据**：
```dart
CommentModel {
  id: string,
  targetId: string,
  targetType: string,
  author: CommentAuthor,
  content: string,
  replyTo?: ReplyTarget,
  replyCount: int,
  likeCount: int,
  isLiked: bool,
  createdAtMs: int,
  isDeleted: bool,
  isPinned: bool,
  isOwn: bool,
  interactionState: CommentIconState
}
```

**后端需求**：
- 验证内容长度和格式
- 检查用户是否被封禁（interactionState）
- 如果是回复，验证 replyToId 是否存在
- 更新父评论的 replyCount
- 返回完整的评论对象（包含作者信息）

#### 2.1.5 切换点赞

**功能描述**：点赞或取消点赞评论

**请求参数**：
```
commentId: string  // 评论 ID
```

**响应数据**：
```
success: bool
newLikeCount: int
isLiked: bool
```

**后端需求**：
- 幂等操作：重复点赞不报错
- 原子更新 likeCount
- 记录用户点赞状态
- 支持高并发

#### 2.1.6 删除评论

**功能描述**：删除自己的评论

**请求参数**：
```
commentId: string  // 评论 ID
```

**响应数据**：
```
success: bool
```

**后端需求**：
- 验证评论所有权
- 软删除：标记 isDeleted = true，保留内容结构
- 不删除子评论（保持树结构完整）
- 更新父评论的 replyCount（可选）

#### 2.1.7 获取子孙数量

**功能描述**：获取某条评论的所有子孙评论数量

**请求参数**：
```
commentId: string  // 评论 ID
```

**响应数据**：
```
count: int  // 子孙数量
```

**后端需求**：
- 使用闭包表快速查询
- 查询条件：`ancestor_id = ? AND depth > 0`
- 需要缓存优化

### 2.3 特殊功能

#### 2.3.1 置顶评论

- 每个目标（帖子/消息）可以有一条置顶评论
- 置顶评论单独返回，不计入普通列表
- 需要权限控制（仅作者或管理员可置顶）

#### 2.3.2 层级限制

- 最多支持 30 层嵌套
- 超过限制时，客户端提示用户使用链接继续讨论
- 后端需要在插入时验证层级深度

#### 2.3.3 已注销用户处理

- 用户注销后，评论保留但作者信息替换为"已注销用户"
- `isDeletedUser = true`
- 头像和用户名使用默认值

## 3. 链接系统 (Link System)

### 3.1 URL 格式规范

#### 3.1.1 基础格式

```
https://lesser.app/{type}/{id}[/subtype/subid]...
```

#### 3.1.2 支持的链接类型

| 类型 | URL 格式 | 说明 |
|------|---------|------|
| 频道 | `/channel/{channelId}` | 频道主页 |
| 消息 | `/channel/{channelId}/message/{messageId}` | 频道消息/帖子 |
| 评论 | `/channel/{channelId}/message/{messageId}/comment/{commentId}` | 评论详情 |
| 锚点 | `/channel/{channelId}/message/{messageId}/anchor/{anchorId}` | 特殊位置（header/bottom） |

## 4. 数据库设计建议

### 4.1 评论表 (comments)

```sql
CREATE TABLE comments (
  id BIGINT PRIMARY KEY,
  target_id BIGINT NOT NULL,
  target_type VARCHAR(50) NOT NULL,
  author_id BIGINT NOT NULL,
  content TEXT NOT NULL,
  reply_to_id BIGINT,
  reply_count INT DEFAULT 0,
  like_count INT DEFAULT 0,
  is_deleted BOOLEAN DEFAULT FALSE,
  is_pinned BOOLEAN DEFAULT FALSE,
  created_at TIMESTAMP NOT NULL,
  updated_at TIMESTAMP NOT NULL,
  
  INDEX idx_target (target_id, target_type, created_at),
  INDEX idx_author (author_id),
  INDEX idx_reply_to (reply_to_id),
  FOREIGN KEY (reply_to_id) REFERENCES comments(id)
);
```

### 4.2 闭包表 (comment_closure)

```sql
CREATE TABLE comment_closure (
  ancestor_id BIGINT NOT NULL,
  descendant_id BIGINT NOT NULL,
  depth INT NOT NULL,
  
  PRIMARY KEY (ancestor_id, descendant_id),
  INDEX idx_descendant_depth (descendant_id, depth),
  INDEX idx_ancestor_depth (ancestor_id, depth),
  
  FOREIGN KEY (ancestor_id) REFERENCES comments(id),
  FOREIGN KEY (descendant_id) REFERENCES comments(id)
);
```

**闭包表说明**：
- 记录所有节点之间的祖先-后代关系
- `depth = 0` 表示自己到自己
- `depth = 1` 表示直接父子关系
- `depth > 1` 表示间接祖先关系

**常用查询**：
```sql
-- 查询某评论的所有子孙数量
SELECT COUNT(*) FROM comment_closure
WHERE ancestor_id = ? AND depth > 0;

-- 查询某评论的根节点
SELECT ancestor_id FROM comment_closure
WHERE descendant_id = ? 
ORDER BY depth DESC LIMIT 1;

-- 查询某评论的所有直接子评论
SELECT c.* FROM comments c
INNER JOIN comment_closure cc ON c.id = cc.descendant_id
WHERE cc.ancestor_id = ? AND cc.depth = 1
ORDER BY c.created_at ASC;
```

### 4.3 点赞表 (comment_likes)

```sql
CREATE TABLE comment_likes (
  comment_id BIGINT NOT NULL,
  user_id BIGINT NOT NULL,
  created_at TIMESTAMP NOT NULL,
  
  PRIMARY KEY (comment_id, user_id),
  INDEX idx_user (user_id),
  
  FOREIGN KEY (comment_id) REFERENCES comments(id),
  FOREIGN KEY (user_id) REFERENCES users(id)
);
```

### 4.4 插入评论时的闭包表维护

```sql
-- 插入新评论
INSERT INTO comments (...) VALUES (...);

-- 插入自己到自己的关系
INSERT INTO comment_closure (ancestor_id, descendant_id, depth)
VALUES (new_comment_id, new_comment_id, 0);

-- 如果是回复，插入祖先关系
INSERT INTO comment_closure (ancestor_id, descendant_id, depth)
SELECT ancestor_id, new_comment_id, depth + 1
FROM comment_closure
WHERE descendant_id = parent_comment_id;
```

## 6. 性能与优化要求

### 6.1 缓存策略

#### 6.1.1 评论入口数据缓存

- **缓存键**：`comment:entry:{targetType}:{targetId}`
- **缓存内容**：`{ avatarUrls: [], count: 0 }`
- **过期时间**：5 分钟
- **更新时机**：新增/删除评论时主动更新

#### 6.1.2 子孙数量缓存

- **缓存键**：`comment:descendants:{commentId}`
- **缓存内容**：`count: int`
- **过期时间**：10 分钟
- **更新时机**：新增回复时主动更新

#### 6.1.3 评论根节点缓存

- **缓存键**：`comment:root:{commentId}`
- **缓存内容**：`rootCommentId: string`
- **过期时间**：永久（评论树结构不变）
- **更新时机**：无需更新

#### 6.1.4 链接元数据缓存

- **缓存键**：`link:metadata:{type}:{id}`
- **缓存内容**：对应的 Info 对象
- **过期时间**：30 分钟
- **更新时机**：内容更新时主动失效

### 6.2 分页性能

#### 6.2.1 游标分页实现

```sql
-- 第一页
SELECT * FROM comments
WHERE target_id = ? AND target_type = ?
ORDER BY created_at ASC
LIMIT 20;

-- 下一页（使用游标）
SELECT * FROM comments
WHERE target_id = ? AND target_type = ?
  AND created_at > ?  -- 游标值
ORDER BY created_at ASC
LIMIT 20;
```

**游标格式**：
- 使用 `created_at` 时间戳作为游标
- 或使用 `id` + `created_at` 组合游标（更精确）

#### 6.2.2 避免深分页

- 禁止使用 `OFFSET`，必须使用游标
- 单次查询限制：最多 100 条
- 总评论数限制：建议不超过 10,000 条（超过时提示用户）

### 6.3 并发控制

#### 6.3.1 点赞并发

- 使用乐观锁或原子操作
- 点赞表使用 `INSERT IGNORE` 或 `ON CONFLICT DO NOTHING`
- likeCount 使用 `UPDATE ... SET like_count = like_count + 1`

#### 6.3.2 回复数并发

- 使用原子操作更新 replyCount
- `UPDATE comments SET reply_count = reply_count + 1 WHERE id = ?`

### 6.4 索引优化

#### 6.4.1 必需索引

```sql
-- 评论表
CREATE INDEX idx_target_time ON comments(target_id, target_type, created_at);
CREATE INDEX idx_reply_to ON comments(reply_to_id);
CREATE INDEX idx_author ON comments(author_id);

-- 闭包表
CREATE INDEX idx_ancestor_depth ON comment_closure(ancestor_id, depth);
CREATE INDEX idx_descendant_depth ON comment_closure(descendant_id, depth);

-- 点赞表
CREATE INDEX idx_user_likes ON comment_likes(user_id, created_at);
```

#### 6.4.2 查询优化

- 评论列表查询：使用 `(target_id, target_type, created_at)` 复合索引
- 子孙数量查询：使用 `(ancestor_id, depth)` 索引
- 根节点查询：使用 `(descendant_id, depth)` 索引

### 6.5 数据量限制

| 项目 | 限制 | 说明 |
|------|------|------|
| 评论内容长度 | 5000 字符 | 超过时截断或拒绝 |
| 评论嵌套层级 | 30 层 | 超过时拒绝插入 |
| 单次查询数量 | 100 条 | 分页限制 |
| 评论总数 | 10,000 条 | 单个目标的评论上限 |
| 头像列表数量 | 5 个 | 评论入口数据 |

## 7. 错误处理

### 7.1 错误码定义

| 错误码 | 说明 | HTTP 状态码 |
|--------|------|------------|
| `COMMENT_NOT_FOUND` | 评论不存在 | 404 |
| `COMMENT_DELETED` | 评论已删除 | 410 |
| `PERMISSION_DENIED` | 无权限操作 | 403 |
| `USER_BANNED` | 用户被封禁 | 403 |
| `CONTENT_TOO_LONG` | 内容过长 | 400 |
| `MAX_DEPTH_EXCEEDED` | 超过最大层级 | 400 |
| `RATE_LIMIT_EXCEEDED` | 操作过于频繁 | 429 |

### 7.2 特殊场景处理

#### 7.2.1 已删除评论

- 返回评论对象，但 `isDeleted = true`
- `content` 替换为"该评论已删除"
- 保留评论结构，允许查看回复

#### 7.2.2 已注销用户

- 作者信息替换为默认值
- `isDeletedUser = true`
- `displayName = "已注销用户"`

#### 7.2.3 评论不存在

- 链接元数据 API 返回 `null`
- 评论详情 API 返回 404
- 客户端显示"内容不存在"

## 8. 安全要求

### 8.1 权限验证

- 发表评论：需要登录
- 删除评论：仅作者或管理员
- 置顶评论：仅目标作者或管理员
- 点赞评论：需要登录

### 8.2 内容审核

- 评论内容需要过滤敏感词
- 支持用户举报功能
- 管理员可以删除任何评论

### 8.3 防刷机制

- 评论频率限制：每分钟最多 10 条
- 点赞频率限制：每秒最多 5 次
- IP 限制：同一 IP 每分钟最多 100 次请求
